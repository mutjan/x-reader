**Phase**: 4

## 🔒 已确定决策（无需再讨论）
### 1. RSS源白名单管理
- **存储方案**: JSON文件
- **存储路径**: `data/sources/sources.json`
- **数据结构**:
  ```json
  {
    "sources": [
      {
        "id": "xxx",
        "name": "源名称",
        "url": "RSS源地址",
        "type": "rss/inoreader/twitter",
        "enabled": true,
        "weight": 1.0, // 源权重，影响评分
        "last_fetched": "2026-04-01T00:00:00",
        "created_at": "2026-04-01T00:00:00"
      }
    ]
  }
  ```
- **功能要求**:
  - 后台页面支持源的增删改查操作
  - 支持启用/禁用源
  - 支持设置源权重（影响新闻评分）
  - 自动备份：每次修改自动备份到`data/sources/backups/`目录，保留30天历史
- **生效机制**: 实时生效，下次抓取时自动使用最新配置

### 2. 内部访问限制
- **实现方案**: 简单密码认证
- **密码存储**: 环境变量配置 `ADMIN_PASSWORD`，使用pydantic-settings管理
- **认证逻辑**:
  - 访问后台页面需要输入统一管理员密码
  - 登录状态通过session保持，有效期24小时
  - 公开前端（GitHub Pages）不受影响，仍可公开访问
- **不需要复杂用户体系**，保持架构轻量，满足内部使用需求

## 🚫 明确不做的范围
- 不引入数据库依赖，保持纯文件架构
- 不实现多用户权限管理，仅用统一密码
- 不修改现有抓取流程的核心逻辑，仅新增源配置管理
- 不影响公开前端的访问权限

## 🎯 成功标准
1. ✅ 管理员可在后台页面管理RSS源（增删改查、启用禁用、设置权重）
2. ✅ 抓取流程自动使用最新的源配置，禁用的源不再抓取
3. ✅ 后台页面仅在输入正确密码后可访问
4. ✅ 公开前端仍可正常访问，不受认证影响
