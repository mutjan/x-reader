# Plan 03-04: 反馈功能迁移到后台管理页面
## Goal
完成反馈功能的落地实现，将原计划的前端反馈功能迁移到后台管理页面，解决GitHub Pages静态环境无法运行后端接口的问题。
## Background
原计划反馈功能在公开前端页面实现，但由于前端部署在GitHub Pages纯静态环境，无法调用后端API接口存储反馈数据。根据最新决策，反馈功能属于内部编辑操作，迁移到后台管理页面实现更合理。
## Dependencies
- 03-02-PLAN.md 已完成反馈存储层`FeedbackStore`的实现
- 后台管理页面`admin.html`和Flask服务`app.py`已存在
## Implementation Steps
### Step 1: 实现后端反馈API接口
修改`src/web/app.py`：
1. 导入`FeedbackStore`类
2. 新增`POST /api/feedback`接口：
   - 验证请求参数：`news_id`必填，`corrected_grade`必须为S/A+/A/B/C，`corrected_score`必须在0-100之间
   - 调用`feedback_store.add_feedback()`存储反馈记录
   - 返回JSON格式的操作结果
3. 接口权限：无需额外权限，后台本身为内部访问
### Step 2: 后台前端新增反馈功能
修改`src/web/templates/admin.html`：
1. 在新闻列表表格的表头新增"操作"列
2. 每条新闻新增"修正评分"按钮
3. 新增反馈模态框，包含：
   - 新闻标题显示
   - 原始评分/分级显示
   - 修正后分级下拉选择
   - 修正后分数输入框
   - 修正原因文本框
4. 实现提交逻辑：调用`/api/feedback`接口，成功后刷新新闻列表
5. 已修正的新闻显示"✓ 人工修正"标记
### Step 3: 移除公开前端的反馈功能
修改`index.html`：
1. 删除反馈弹窗`feedbackModal`相关HTML代码
2. 删除`openFeedback()`、`closeFeedback()`、`submitFeedback()`函数
3. 删除每条新闻评级标签旁的"修正"按钮↻
4. 移除相关的CSS样式（如果有单独的反馈相关样式）
### Step 4: 集成反馈到发布流程
修改`src/publishers/github_pages.py`：
1. 导入`FeedbackStore`类
2. 在生成发布数据前加载所有反馈记录
3. 对有反馈记录的新闻，使用`corrected_grade`和`corrected_score`覆盖原始AI评分
4. 新增`is_corrected: true`字段标记人工修正的新闻
5. 发布到GitHub Pages的数据中包含修正后的评分和标记
## Success Criteria
- [ ] 后台新闻列表每条新闻都有"修正评分"按钮
- [ ] 点击按钮弹出反馈表单，提交后数据成功保存到`data/feedback/feedback_records.json`
- [ ] 提交反馈后页面刷新，显示修正后的评分和人工标记
- [ ] 公开前端页面没有任何反馈相关的元素和代码
- [ ] 执行发布脚本后，有反馈的新闻在公开页面显示修正后的评分和✓标记
- [ ] 反馈数据自动备份功能正常工作
## Risk Mitigation
- 确保API接口参数验证完整，避免无效数据写入
- 发布流程中反馈数据覆盖逻辑要确保不会修改原始数据，仅在发布时临时替换
- 移除前端反馈代码时不要影响其他功能的正常运行
