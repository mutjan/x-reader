# Phase 3: AI Enhancement & Calibration - Context Decisions
**Phase**: 3
**Last updated**: 2026-04-01
**Status**: 决策已锁定，可进入规划阶段

## 🔒 已确定决策（无需再讨论）
### 1. 同事件新闻自动分组
- **实现方案**: 混合模式
- **具体逻辑**:
  1. 第一阶段：基于实体匹配快速过滤，拥有3个及以上相同核心实体的新闻进入候选集
  2. 第二阶段：对候选集使用标题+摘要的语义相似度计算，相似度阈值≥0.85判定为同事件
  3. 第三阶段：按发布时间排序生成事件时间线，保留最新/最完整的新闻作为主条目
- **不需要额外引入外部embedding服务**，优先使用现有规则和简单字符串相似度算法实现，保持架构轻量

### 2. 人工评分反馈存储
- **存储方案**: 独立JSON文件
- **存储路径**: `data/feedback/feedback_records.json`
- **数据结构**:
  ```json
  {
    "records": [
      {
        "news_id": "xxx",
        "original_grade": "A",
        "original_score": 80,
        "corrected_grade": "S",
        "corrected_score": 92,
        "reason": "重大行业事件，符合S级标准",
        "timestamp": "2026-04-01T12:00:00",
        "entities": ["OpenAI", "GPT"]
      }
    ]
  }
  ```
- **自动备份**: 每次修改自动备份到`data/feedback/backups/`目录，保留30天历史
- **实现位置调整**: 反馈功能从公开前端页面迁移到后台管理页面实现
  - ✅ 原因：公开前端部署在GitHub Pages纯静态环境，无法运行后端API
  - ✅ 访问控制：评分修正属于内部编辑操作，仅在后台对授权用户开放
  - ✅ 接口实现：在Flask后台服务新增`POST /api/feedback`接口处理提交

### 3. 评分校准逻辑
- **生效时机**: 离线批量校准
- **执行频率**: 每日凌晨自动执行一次
- **校准逻辑**:
  1. 基于反馈记录统计不同实体、类型、关键词的评分偏差
  2. 生成校准规则（如：包含"OpenAI"且类型为"product"的新闻基础分+3）
  3. 重新计算过去7天内所有新闻的评分和分级
  4. 规则保留最近30天，过期自动失效
- **不做实时校准**，避免评分波动影响内容稳定性

## 🚫 明确不做的范围
- 不引入外部向量数据库或embedding服务
- 不实现实时反馈校准功能
- 不引入数据库依赖，保持纯文件架构
- 不修改现有Phase 2的核心处理流程，所有增强功能为增量添加
